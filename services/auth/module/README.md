# Messenger auth skeleton

This batch contains only the database/configuration boundary for a small OAuth-style auth service. Registration, login, refresh endpoints, password hashing, JWT encoding, and Vault calls are intentionally not implemented yet.

## Current models

- `User`: email, password hash, one simple global role, active flag, token version.
- `DeviceSession`: one browser/app installation; its UUID can be placed in the JWT `sid` claim.
- `RefreshToken`: stores only a token hash and rotation-family metadata.
- `SigningKey`: stores public metadata plus a backend-specific key reference, never private key material.

## Local versus Vault signing

Access tokens keep the same JWT/OAuth shape in both modes. Only the private signing operation changes:

- `local`: development/testing signer using a private key from a protected external source.
- `vault`: production-oriented signer delegating signing to Vault Transit.

Refresh tokens are opaque random values in both modes. They are not Vault-signed JWTs; only their hashes are stored.

## Roles

The first version gives each user exactly one global role: `user`, `moderator`, or `admin`. This is intentionally smaller than a full role/permission schema. Conversation membership and ownership checks should later live in the messenger module, because a global role cannot decide access to a particular chat.

## Run locally

```bash
cp .env.example .env
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

docker compose up -d pgsql
alembic upgrade head
uvicorn app.main:app --reload
```

Health check:

```text
GET http://localhost:8000/health
```

## Alembic

Alembic imports `app.models`, so all tables are registered in `Base.metadata` and autogeneration works:

```bash
alembic revision --autogenerate -m "describe change"
alembic upgrade head
```

Always review autogenerated migrations before applying them.
