## Модель аутентификации и авторизации 

- **L1 — stateless**: проверка только JWT (подпись/срок/claims), без обращения к хранилищу сессий.
- **L2 — stateful (step-up)**: для критических действий дополнительно проверяется состояние серверной сессии и “недавняя аутентификация”.

---

## 1) Общее устройство модели

### 1.1. Принцип
1. При логине сервер создаёт **сессию устройства** и выдаёт клиенту пару токенов (**ACCESS** + **REFRESH**).
2. Для большинства API-запросов используется только **ACCESS** (L1). Сервер **не** обращается к хранилищу сессий.
3. Для продления работы используется **/auth/refresh** с **REFRESH** (stateful проверка активной сессии устройства и ротация refresh-токена).
4. Для критических действий требуется **step-up** до L2: дополнительная проверка OTP и окно.

### 1.2. TTL плюс минус
- **ACCESS**: 5 минут.
- **REFRESH**: 30 дней скользящий срок при активности.
- **recent_auth_until** (L2-окно): 5 минут.

---

## 2) Логин и создание сессии устройства

### 2.1. Триггер
Вход будет выполняться через пароль + OTP.

### 2.2. Действия сервера при успешном входе
Сервер:
1. создаёт запись **device session** для данного `device_id` (генерирует `sid`);
2. выпускает **ACCESS** и **REFRESH**, связанные с этой сессией;
3. возвращает токены клиенту.

> Важно: “L1-сессия” в контексте этого документа означает, что сессия устройства создана на сервере, но для L1-запросов она **не** проверяется.

### 2.3. Набор claims ACCESS JWT плюсминус
- `sub` — user id
- `exp` —  срок действия
- `scope` — права/роли
- `sid` — session id (device session)
- `iat` — время выдачи 

---

## 3) Обычные API-запросы (L1 — stateless)

На L1 сервер:
1. проверяет подпись JWT и `exp`;
2. извлекает claims (`sub`, `sid`, `scope`);
3. выполняет **авторизацию** на основе `scope` и правил доступа к ресурсу;

---

## 4) Обновление токенов (Token Refresh)

### 4.1. Когда выполняется refresh
Когда **ACCESS** истёк (или близок к истечению), клиент вызывает:
- `POST /auth/refresh` с **REFRESH**.

### 4.2. Серверная логика refresh (stateful)
Сервер обязан:
1. убедиться, что refresh-токен относится к **активной** device session (`sid`) и не отозван;
2. выполнить **ротацию** refresh-токена:
   - пометить старый refresh как недействительный;
   - выдать **новый REFRESH** и **новый ACCESS**;
3. фиксировать `last_seen_at` и  метаданные сессии `IP + UA + версия клиента`.

### 4.3. Повторное использование старого refresh (reuse detection)
Если сервер получает refresh-токен, который уже был использован/инвалидирован:
- **немедленно отозвать** соответствующую device session,
- вернуть ошибку, сигнализирующую клиенту о необходимости полного перелогина.

> Это ключевое свойство безопасности: refresh-ротация + детект повторного использования.

---

## 5) Критические действия (L2 — stateful step-up)

### 5.1. Что считается критическим
- изменение пароля,
- удаление аккаунта,
- отзыв всех сессий

### 5.2. Как выполняется step-up
1. Клиент проходит дополнительную проверку OTP(email).
2. Сервер обновляет поле `recent_auth_until` у device session (`sid`) на текущий момент + 5 минут.
3. Критические endpoints принимаются только при активном `recent_auth_until`.

### 5.3. Поведение при отсутствии L2-окна
Если `recent_auth_until` неактивен:
- сервер возвращает доменную ошибку **STEP_UP_REQUIRED**,
- клиент инициирует step-up-флоу и повторяет запрос.

---

## 6) Управление устройствами и сессиями

Управление устройствами строится вокруг **stateful device sessions**.

### 6.1. Операции
- **Просмотр активных сессий**: список устройств/сессий пользователя (с метаданными: модель, платформа, last_seen, created_at).
- **Отзыв конкретного устройства**: revoke по `sid` (немедленно блокирует refresh и L2, а L1 “дотухает” по TTL ACCESS).
- **Отзыв всех сессий**: revoke всех `sid` пользователя; операция требует **L2**.

### 6.2. Эффекты отзыва
При revoke конкретной сессии:
- refresh становится непригодным (обновление токенов невозможно),
- L2-доступ закрывается немедленно,
- L1-доступ сохраняется лишь до истечения уже выданного ACCESS.

---

## 7) Рекомендации по реализации (кратко)

### 7.1. Хранилище сессий
Достаточный минимум для записи device session:
- `sid`, `sub`, `device_id`
- статус: `active/revoked`
- `refresh_token_hash` (хранить **хеш**, не сырой токен)
- `created_at`, `last_seen_at`
- `recent_auth_until`
- опционально: `ip_last`, `ua_last`, `client_version`

### 7.2. Хранение токенов на клиенте
- ACCESS можно хранить в памяти и перевыпускать через refresh.
- REFRESH хранить в защищённом хранилище платформы (Keychain/Keystore/secure storage).
- Никогда не логировать токены.

### 7.3. Коды ошибок (рекомендуемая семантика)
- `TOKEN_EXPIRED` — истёк ACCESS
- `INVALID_TOKEN` — неверная подпись/формат
- `SESSION_REVOKED` — сессия устройства отозвана (обычно выявляется на refresh/L2)
- `STEP_UP_REQUIRED` — требуется L2
- `REFRESH_REUSE_DETECTED` — повторное использование refresh (требуется перелогин)

## 8) Краткие последовательности (для согласования)

### 8.1. Login
Client → Server: login pass+otp  
Server → Client: ACCESS + REFRESH, create device session(sid)

### 8.2. L1 API call
Client → Server: Authorization: Bearer ACCESS  
Server: verify JWT (no session store) → authorize by scope

### 8.3. Refresh rotation
Client → Server: POST /auth/refresh (REFRESH)  
Server: check active sid → invalidate old refresh → issue new refresh + access

### 8.4. Step-up (L2)
Client → Server: step-up otp  
Server: set recent_auth_until = now + 5m  
Client → Server: critical request  
Server: allow only if recent_auth_until active
